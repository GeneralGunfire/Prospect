22 March 2026

Research and Data Foundation

No code written. The first priority was the data the platform would run on — bursaries, careers, and a clear picture of the South African edtech landscape.
Bursary Database

South African bursary information is scattered across university websites, government departments, and private companies, all published in different formats with inconsistent detail. The initial migration pulled 200 bursaries into a consistent data structure: name, field of study, eligibility criteria, funding amount, application deadline, and which provinces the bursary applied to. From there, 19 more were added, then another 26, bringing the total to 245 entries.

Getting the fields consistent was more work than the numbers suggest. Deadline formats varied — some entries listed a calendar date, others said "rolling" or "contact the institution." Province lists were written differently across sources. Eligibility descriptions used different terminology for the same requirements. Everything had to be normalized by hand so the search and filter features would actually work.

A document called BURSARY-REVIEW.md was created as an audit trail, tracking which entries had gaps and which had been verified. This mattered because the data would be shown directly to students — an incomplete or misleading entry is worse than no entry at all.

Career Data Cleaning

Two duplicate career entries were found during an import check — one for AI/ML Engineer and one for Solar Installer, both added twice through different import passes. These were removed before the data was used anywhere in the app.

Competitor Research

The rest of the day covered market research: looking at what existed in the South African edtech space — Yenza, Gradesmatch, NCAP, Reslocate — what each did well, and where students were being underserved. The analysis found a specific gap: most platforms focused solely on university applications or provided generic career advice without connecting it to the subject choices students were actually making in Grades 10–12. A 7-phase project roadmap was drafted from these findings and the v1 feature requirements were written down. The product direction was clear from the outset: make the connection between subject choice and career outcome visible and concrete.

25 March 2026

Project Scaffold and Planning

Project configuration initialized. Architecture decisions locked in for shared state management and folder structure.
Phase 0 Plan

The project configuration was initialized and a Phase 0 plan was written covering how the app scaffold would be structured. The first architecture decision was how shared state across the application would work. The choice was React context rather than a more complex state management library like Redux. The reasoning: the app was small enough that a simpler approach worked fine, and introducing Redux would add overhead in file size and developer complexity without paying for itself at this scale. React context handles auth state, user preferences, and navigation state cleanly without additional tooling.

Folder Structure

The project folder structure was established to support the three major sections the app would eventually have: school-related features, career features, and community features. Getting this right early was intended to avoid the reorganization pain that comes from adding features to a flat folder structure for months and then having to sort it out later. A partial reorganization still happened in May, but having a domain-based structure from the start reduced how much had to be moved.

Roadmap

The 7-phase roadmap drafted on 22 March was formalized into planning documentation covering what each phase would deliver, the dependencies between phases, and the success criteria for each. Having this written down made it possible to track progress and make deliberate decisions about what to build next rather than reacting to whatever seemed most urgent on a given day.

27 March 2026

First Visual Pass — Landing Page and Auth

First working UI. Black and white palette with a translucent navbar. Signup wired up. Error messaging for rate-limited auth fixed immediately.
Landing Page — First Build

The landing page was built for the first time. The initial design used black and white as the primary colors, with a translucent navbar that blurred content behind it on scroll. It was a starting point — it would be rebuilt entirely the following day when the framework decision changed — but it established the visual direction: clean, high-contrast, text-forward.

Signup Page and Error Handling

The signup page was wired up with form validation. During testing, an issue appeared immediately. When the same email address is submitted too many times in quick succession — which happens constantly during development — Supabase's auth service rate-limits the request and returns an error. The message being shown to the user was raw technical text: something like "Email rate limit exceeded." That's meaningless to a student who just tried to create an account. This was fixed to show a human-readable message explaining what happened and what to do next. A small fix, but the kind that determines whether a user retries or leaves.

Hero Section Redesign

The hero section was redesigned to take up more of the screen. The original version was modest in size and didn't communicate the platform's purpose on first glance. For a platform competing with established edtech products, the landing page has to earn attention in the first few seconds — the hero is where that happens.

28 March 2026

Framework Switch and Landing Page Rebuild

Next.js scrapped. Rebuilt on React and Vite. Premium animations added. Data saver toggle introduced for users on limited mobile data.
Why Next.js Was Dropped

The project was initially built in Next.js. This was scrapped on 28 March. The reason was straightforward: Prospect is a client-side application. None of the pages needed server-side rendering, there were no SEO-critical dynamic routes that required pre-rendering at request time, and there was no backend API layer that made sense to host as Next.js API routes. Next.js adds real complexity around build configuration, routing conventions, hydration behavior, and deployment — none of which was paying for itself here. The switch was made to a standard React and Vite setup: simpler to configure, faster to build, and straightforward to deploy to Vercel.

Landing Page Rebuild

The landing page was rebuilt from scratch on the new foundation. This rebuild introduced premium animations that the first version didn't have: glassmorphism effects (translucent panels with backdrop blur over content), a logo animation that ran on page load, and entrance animations for each section. The visual direction became more premium — the kind of design that signals the platform is worth spending time with.

Data Saver Toggle

A data saver toggle was added — a control that let users reduce or disable animations and heavy effects. This was a deliberate decision given the South African context. Mobile data costs in South Africa are among the highest in the world relative to income, and many students browse on limited prepaid bundles. A landing page full of glassmorphism and entrance animations can be beautiful on WiFi and painful on 50MB of remaining data. The toggle let users make that call for themselves.

30 March 2026

Core App Features Launch

First full working React application. Study library functional. Data persistence solved with localStorage. Subject selector built. Navigation bugs fixed.
Study Library

The study library was cleaned up and made functional. Before this it existed as a skeleton — pages were there but content and navigation weren't working correctly. Study material was now accessible and browsable for the first time.

Data Persistence — LocalStorage

Data persistence was the most significant problem solved on this day. User progress, bookmarks, and quiz results were all being stored in memory only, which meant they disappeared on every page refresh. For an educational app where students need to pick up where they left off, this was a fundamental failure. LocalStorage was introduced as the storage layer: every piece of user data that needed to survive a page reload was written there on change and read back on app initialization.

Subject Selector

The Subject Selector for Grade 10 was implemented. A student could select the subjects they were taking and the app would surface which careers those subjects made accessible. This was one of the core value propositions of the platform — making the connection between subject choice and career outcome visible and concrete rather than abstract.

Navigation and Loading Animation

Several navigation problems were fixed. The hamburger menu on mobile wasn't opening and closing correctly. There were z-index conflicts causing the navigation bar to appear behind other elements on certain pages. A back button was added on the map page, which had no obvious way to return from it. The loading animation went through multiple iterations before settling on a 5-second display on app load — longer than typical, but chosen deliberately to give the app's initial data load time to complete before rendering the dashboard, avoiding a flash of empty content.

31 March 2026

Polish and Accessibility

Full accessibility audit. Font size violations corrected. Active page indicator added to the header. Animation stagger delays capped for long lists.
Accessibility Audit

A full accessibility review was run across the entire app. The most common violation was font sizes — several text elements were below a comfortable readable size. These were corrected across all pages. Touch target sizes were also verified to meet the 48px minimum required for reliable mobile use.

Active Page Indicator

The header was updated with an active page indicator so users could see at a glance which section they were in. This had been missing from the first build. Without it, navigating between sections left users with no visual confirmation of where they'd landed.

Animation Stagger Cap

Animation stagger delays were capped across the app. The way stagger animations work is that each item in a list is delayed slightly more than the previous one, creating a cascading reveal effect. On short lists this looks good. On longer lists — like the bursary list with 245 entries — the accumulated delay was still running several seconds after the page had fully loaded because items near the end of the list had delays that stacked to unreasonable lengths. A cap was introduced so that no item waited beyond a fixed maximum regardless of its position in the list.

3–7 April 2026

Component Library, Architecture, and Bug Triage

11 UI components integrated. Bookmark persistence fixed. TVET section expanded. Map page blank-screen traced and resolved. Event scheduler crash found.
Component Library

Eleven UI components were integrated into a shared design system over these days. Cards, buttons, and form elements were standardized so they looked and behaved consistently across the app. Before this, each section had been styled independently and the inconsistencies had started to accumulate.

Bookmark Persistence Fix

Bookmark persistence was still broken in some cases despite the localStorage work on 30 March. The issue was a timing problem: a read would happen before a write had completed, causing the app to treat bookmarks as if they didn't exist. Fixed with a more explicit read/write pattern that ensured writes resolved before any dependent reads occurred.

TVET Section Expansion

The TVET (Technical and Vocational Education and Training) section was expanded significantly. TVET colleges represent a major pathway for South African students who won't attend a traditional university — they offer practical qualifications in engineering, business, IT, hospitality, and dozens of other fields. The platform had underrepresented this pathway in its initial data.

Map Page Bug — Blank Screen

A critical bug was found: selecting a location on the map page caused the page to go completely blank. The root cause was a routing state issue — the component wasn't handling the state transition correctly after a selection, causing it to render nothing. Fixing this also required restoring a heatmap animation accidentally removed during an earlier edit. The footer was also overflowing on small screens and was corrected.

Event Scheduler Crash — 7 April

A crash was found in the event scheduler. The cause was a dropdown component receiving an empty string as the value for one of its items. The component library in use (Radix UI) does not accept empty strings as item values — it requires either a real value or explicit null. The component was silently failing when it encountered the empty string during render. The fix was replacing the empty string with a proper placeholder value.

15–17 April 2026

Product Rethink, Restructure, and Dashboard Rebuild

Major restructure into 3 pillars. Landing page rebuilt to reflect them. Dashboard rebuilt from feature directory to daily-focus view. Community Impact feature built. 30 tests written and passing.
Three Pillars — 15 April

A significant rethink happened about how the product was organized. The original structure had everything loosely connected. The decision was made to separate the app into three distinct pillars: School Assist (study tools and guidance), Career Guide (career exploration and the quiz), and Community (civic awareness and utilities). The landing page was rebuilt to communicate these three pillars clearly. Routing was updated to reflect the new organization and navigation that had accumulated inconsistencies over the previous weeks was fixed throughout.

Dashboard Rebuild — 17 April

The dashboard was completely rebuilt. The original version was essentially a feature directory — it told you what the app contained and let you navigate to different sections. That is not particularly useful once a user knows the app. The new dashboard was rebuilt as an actionable daily-focus view, surfacing information relevant to where the student was in the school year: next deadline, next subject to study, recommended session. Everything else was secondary to that primary block.

Community Impact Feature — 17 April

Community Impact was built as a new feature: a page where students could share how Prospect had helped them, with a public submissions feed and aggregate stats. The first version had authentication stubs that didn't actually talk to the database — this was caught when a submission appeared to succeed but nothing was written anywhere. Real authentication was connected and the submission flow was verified end to end. A navigation error was also found: the page had been accidentally included inside the school navigation menu, which didn't make contextual sense. It was moved to a standalone page.

Testing

30 end-to-end tests were written covering the dashboard, community impact, and authentication pages. All 30 were passing by end of day. This was the first meaningful test coverage the project had.

18 April 2026

Reverts After Visual Overhaul

The full visual redesign from 17 April broke three areas. Selective revert applied. Images compressed to WebP. TVET hero improved with diagonal gradient.
What Broke

The visual redesign attempted on 17 April — design tokens, a component library overhaul, consistent spacing and color applied everywhere — caused problems in three places: the school assist pages had their layouts disrupted, the quiz page broke entirely, and the career card design was changed in a way that conflicted with the existing data structure for career information.

The Decision to Revert

Rather than patching each issue individually, those three areas were reverted to their state before the redesign. The landing page changes were kept because they were working correctly. This was the right call. Trying to patch multiple breaking changes simultaneously tends to produce more breaking changes, and the cost of selective reverts is always lower than the cost of debugging a cascade of failures introduced all at once.

Image Compression

All images across the site were compressed to WebP format, with a fallback for browsers that didn't support it. The reduction in page load weight was meaningful on mobile — PNG and JPEG files that had been several hundred kilobytes were replaced with WebP equivalents a fraction of that size, with no visible quality difference.

TVET Hero

The TVET section hero background was improved with a diagonal gradient. It had been using a flat color that felt static compared to the rest of the landing page's visual language.

19 April 2026

Persistent Login and School Assist Improvements

Users now stay logged in across tabs and sessions. School Assist search upgraded to dual-mode. Pothole reporting and water dashboard started.
Persistent Login

Persistent login was implemented so users stayed signed in across browser tabs and after closing and reopening the browser. Before this, closing a tab would sign a user out — frustrating for students using the platform across multiple study sessions. Supabase handles session persistence natively when configured correctly. The fix was ensuring session refresh logic ran on tab focus and app initialization, not only on explicit login actions.

School Assist Search

The School Assist section got better search — a dual-mode search that could find study library topics by name and also return relevant content snippets from within topics. This was added to the navigation bar for the school assist section so it was accessible from any page within that section without needing to navigate to the library first.

Pothole Reporting — Started

A pothole reporting feature was started as part of the Community section. Users could flag potholes by location, view other reports, and track whether they'd been fixed. The backend was connected to Supabase with proper authentication. The location input originally tried to use GPS — browser geolocation — but this wasn't working reliably across different devices. The input was replaced with a hierarchical dropdown: Province, then City, then Suburb, then a free-text Street field. More reliable and easier to use than asking someone to share GPS coordinates.

Water Dashboard — Started

The water dashboard was started: a page showing dam levels by province, active water alerts, and maintenance updates. The initial version used static placeholder data, which would later be replaced with live scraped data via GitHub Actions.

20–21 April 2026

Career Data Expansion to 400 Careers

Major data build. Every career given a full South African profile. Small set verified first, then expanded to 400 total.
What Changed

The career cards were redesigned to be substantially more informative. The previous version showed a title, sector, and a brief description. The new version gave each career a full South African profile: salary ranges specific to South Africa (not imported from US or UK sources), job demand by province, required subjects and minimum APS scores, what a student in Grades 10, 11, and 12 should be doing to work toward that career, and real South African examples — companies hiring in that field, internship programs, and bursaries specific to that discipline.

Build Approach

The approach was to build a small set of careers thoroughly first, wire them up, and verify the data was displaying correctly before expanding. This avoided the situation where a structural problem with the data format only appeared after 400 entries had already been written to that format. Once the structure was confirmed — the right fields showing, filtering working, detail pages rendering correctly — the remaining careers were updated to the same standard.

Result

The final count reached 400 careers. Every entry had comprehensive South African-specific data rather than generic descriptions. Routing and card presentation were verified working across the full dataset before moving to the next phase of development.

25–27 April 2026

Bug Fixes, New Pages, and Navigation Overhaul

Quiz completion fixed. Animated loading screen added. Login pages redesigned to match landing page. Civics, Tax calculator, Cost of Living, and Chat page added.
Bug Fixes — 25 April

A dedicated session cleared a backlog of persistent bugs. The quiz completion button wasn't working correctly in some states — the final question could be answered but the results page wasn't triggering. Career cards rebuilt in the data expansion needed to replace the old routing, which still pointed to the previous card format. Various mobile layout issues were addressed across multiple pages.

Loading Screen and Landing Page — 26 April

A loading screen with animated morphing text was added: the words Discover, Plan, Elevate, and Succeed cycling through on app load. The first version had a bug — it only showed the first two words before ending because the timing logic didn't account for the full cycle. Several iterations were needed. The login and signup pages were redesigned to match the landing page's color scheme — they had been using their own unrelated styling that made the transition from landing to auth feel like a different product. The navbar spinning animation when opening the mobile menu was also fixed; it had been rotating in a way that looked like a rendering error.

New Pages — 27 April

Four new pages were added: the Civics page (a guide to South African civics relevant to students), the Tax and Budget Calculator, the Cost of Living tracker, and the School Assist Chat page. Routing was updated for all four. The chat page was a shell at this point — the structure was there but the AI backend wasn't wired yet. The dashboard was cleaned up to remove sections showing placeholder data, and the navigation bar was updated to use the same design as the landing page across all feature sections. A color palette audit was also run — the site had accumulated too many stray colors from different sessions and everything was brought back to blacks, greys, and whites.

1–3 May 2026

Learning Modules and Study Library Consolidation

Two Grade 10 Algebra modules built with mastery tracking. Study library consolidated into a single location. Progress tracking bug fixed. Water dashboard redesigned with province selector.
Grade 10 Algebra Modules

Two interactive learning modules were built for Grade 10 Term 1: Introduction to Equations and Simultaneous Linear Equations. These were structured as guided lessons with worked examples, practice questions, and mastery tracking. When a student completed a topic, that progress was recorded in Supabase so it persisted between sessions — a student could close the browser, come back the next day, and see which topics they'd already covered.

Study Library Consolidation

The study library was consolidated so all learning material lived in one place. Content had accumulated in different locations as features were added across March and April — some material accessible through one navigation path, some through another. Pulling everything into one unified section cleaned up both the codebase and the user experience.

Progress Tracking Bug

A bug appeared in progress tracking during testing: the mastery state for the first algebra topic was displaying correctly in the library but not on the dashboard. The dashboard was showing the topic as incomplete even after it had been marked complete. The cause was a data-fetching timing issue — the dashboard was reading from local state before the database response had returned, so it was rendering stale data. Fixed by making the dashboard wait for the correct data source before rendering the progress section.

Mobile Display and Water Dashboard

The algebra content had a display issue on mobile: equations that were too long for a small screen would wrap mid-expression in a way that made them unreadable. The equations were reformatted to handle small screens gracefully without reducing text size. On 2 May, the water dashboard was redesigned — the original showed all provinces simultaneously in one crowded chart, replaced with a province selector so users could focus on one at a time. On 3 May, a broad polish pass was run across all pages: spacing, typography, card padding, layout rhythm. The kind of work that makes an app feel finished rather than assembled.

7–8 May 2026

Testing, Code Cleanup, and Pothole Improvements

Pothole feature improved with fake report detection. Ambiguous database column error traced and fixed. Codebase reorganized by domain. All 31 end-to-end tests confirmed passing.
Pothole Feature Improvements

The pothole reporting feature received improvements on 7 May: a fake report detection system and a user-flagging mechanism so other users could mark reports they believed were inaccurate. Flagged reports were queued for review before being shown prominently. This was a reasonable approach to the spam problem without requiring manual moderation of every submission.

Database Error — Ambiguous Column

A database error appeared during development: an ambiguous column reference after a schema update. A query was referencing a column name that existed in multiple tables being joined, and the database engine couldn't determine which table's version was intended. Fixed by making all column references in the query fully qualified — specifying both the table name and the column name for every field in the select and where clauses.

Codebase Reorganization — 8 May

A complete codebase audit was run. Pages and components added to a flat folder structure over two months were reorganized into subfolders by feature domain: school, community, careers, home. Dead code, stale routes, and files left over from earlier experiments were removed. The flat structure had made it difficult to find files and had resulted in some components being created twice in different locations. All 31 end-to-end tests were confirmed passing after the reorganization — the reorganization had changed import paths in several places and these were updated and verified with Playwright before moving on.

11 May 2026

Community Impact Removed, Declutter Pass

Community Impact removed after reassessing its value for the core audience. School assist and calendar decluttered on mobile. FEATURES.md and CODEBASE_MAP.md written.
Community Impact Removed

The Community Impact feature — built in April as a page where students could share how Prospect had helped them — was removed on this date. The reasoning was about audience fit. Students in Grades 10–12 are not the primary audience for a community impact reporting tool. The feature required its own authenticated submission flow, its own database tables, and its own maintenance. That overhead wasn't justified by the value it actually provided to the core user. Removing it simplified the platform and made the remaining features easier to maintain and explain to new users.

Mobile Declutter — School Assist and Calendar

The school assist dashboard and calendar pages were decluttered on mobile. Both had accumulated content across multiple development sessions and were showing too much information in too small a space on phone-sized screens. Elements that weren't critical to the primary task on each page were removed or collapsed into secondary views accessible by interaction, rather than being displayed by default.

Documentation

FEATURES.md and CODEBASE_MAP.md were written to document what existed in the project and how the codebase was organized. After two months of development, the reasoning behind some architectural decisions wasn't obvious from the code alone, and several features had been added and removed, leaving traces in various places. Having written documentation made it possible to get a clear picture of the platform without having to read every file.

13 May 2026

Pothole Feature Removed, Focus Sharpened

Pothole reporting removed — outside the platform's focus. Project structure cleaned. Full UI polish session. Gemini API confirmed broken with 429 rate limit error.
Pothole Reporting Removed

The pothole reporting feature, built in April and improved in early May, was removed on 13 May. The reasoning was product focus. Prospect is a career and education guidance platform for high school students. A pothole reporting tool is a civic utility — unrelated to careers, education, or the decisions a Grade 10–12 student is actually trying to make. Keeping it in the platform made Prospect harder to explain and harder to develop with a clear direction. The platform had civic content that made sense because it directly affected students: cost of living data helps with financial planning, load shedding tracking helps plan study sessions, exam dates are directly relevant. Pothole reporting didn't meet that standard and was removed.

Project Structure Cleanup

Duplicate files, leftover planning documents from earlier phases, and stale markdown files that had accumulated over the development period were cleared out alongside the removal.

UI Polish Session

A full UI improvement session was run using the Impeccable design framework — a structured approach to reviewing spacing, font sizing, layout rhythm, and color consistency. The session reviewed every feature section against the landing page's established visual language. The chat feature had a typing bug found during this session: the text input wasn't responding correctly to keystrokes in some states, causing characters not to register. This was fixed.

Gemini API — Confirmed Broken

The Gemini API was tested to check if the chat's AI backend would work on the live site. The test returned a 429 rate limit error — the free tier quota for the API key in use had been exhausted. The chat's AI connection was effectively non-functional on the live site. This would need to be resolved before launch.

14 May 2026

Community Section Rebuilt and Routing Fixed

Community rebuilt with load shedding tracker and NSC exam countdown. Routing problems from the rebuild fixed systematically. Career quiz and TVET pages improved.
Community Section Rebuild

The community section was properly structured. A CommunityPage hub served as the entry point with links to each community feature. Two new pages were the main additions.

LoadSheddingPage: a real-time electricity stage tracker pulling live Eskom stage data. The direct use case for students is planning study sessions around power outages — if Stage 4 load shedding is scheduled from 18:00 to 20:30, a student can plan accordingly. A specifically South African problem that the platform is well-positioned to address.

MatricExamDatesPage: the full 2026 NSC exam timetable with live countdown timers showing days remaining until each exam period. Matric exams are the most high-stakes event in a Grade 12 student's year. Having the exact schedule and a live countdown accessible in the same platform where they're studying makes the connection between daily study and exam preparation concrete.

Routing Problems From the Rebuild

The rebuild caused a set of routing problems. The community link in the navigation bar returned a blank page after the restructure. The auth system broke in places. The school assist section briefly lost its navigation bar entry. These were fixed systematically, with navigation across the entire site verified end to end before continuing.

Career and TVET Improvements

The career quiz page was improved with better typography and tighter spacing. Career cards got an expanded layout when clicked, showing more detail inline rather than requiring a full page navigation. The TVET section's navigation was identified as inconsistent — clicking between sub-pages wasn't behaving predictably — and was addressed in the same session.

15 May 2026

AI Integration, Data Scrapers, Final Polish, and Launch

Automated scrapers deployed for live data. Three AI backends tried: Gemini failed, Ollama worked locally but can't serve cloud users, Groq deployed and verified working. Site launched.
Automated Data Scrapers

Scrapers were set up for both the load shedding page and the water dashboard, running on a schedule via GitHub Actions. They pull live data from public sources and commit updated JSON files to the repository. Vercel picks up the new commits and redeploys automatically, so the data stays current without manual intervention. The water dashboard had been using static placeholder data since April — this replaced it with real, current figures.

AI Chat Backend — Three Attempts
Attempt 1 — Google Gemini
Failed — Rate Limited

An API key had been configured and the chat was built to call Gemini. When tested on the live site, it returned a 429 error — the free tier quota had been exhausted between development testing and live site traffic. Rather than leaving the chat broken, a predefined answer system was built immediately as a fallback: seven detailed answers to the most common student questions, stored locally in the app, returning instantly without any network call. This made the chat feel functional for the most common questions even without a working AI connection.

Attempt 2 — Ollama (Local AI)
Failed — Local Only

Ollama is a tool for running AI models on your own machine. The llama3.2 model (2GB) worked correctly during development — fast and accurate. A second model called llava (4.7GB) was also downloaded to handle image uploads. Both worked perfectly locally. The problem appeared on the live site: Ollama runs as a local server on localhost. Every user visiting the site was sending requests to localhost on their own machine, which wasn't running Ollama. Every request failed. Ollama is a local development tool, not a cloud service, and cannot be used as a backend for a hosted web application.

Attempt 3 — Groq
Working

Groq provides free cloud inference for llama3 models and is notably fast. An API key was created, added to the project's local environment variables and to Vercel's environment configuration for production. The chat was updated to call Groq's API using the llama-3.3-70b-versatile model. The predefined answers were kept as the first response layer — for the seven common questions, the answer still appeared instantly from the local store, with Groq silently upgrading it in the background if its response differed. Any question outside those seven was sent directly to Groq. The site was redeployed and the chat verified working end to end on the live URL.

Final Design Pass and Launch

The remaining pages that hadn't received a thorough design pass — TVET, library, calendar, and community pages — were reviewed and improved. Auth was removed from the informational community pages since they contain no user-specific data and requiring login to view them served no purpose. The Grade 10 Subject Selector was removed from the landing page navigation — it had been listed as a feature entry point but was never fully built to a shippable standard and was causing confusion. The site was redeployed with all changes in place and verified working across all major features before being considered launched.
