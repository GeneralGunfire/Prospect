Methodology — Section 1
01

Design Criteria

Before any code was written, a set of design criteria were established to keep decisions grounded in the research rather than in personal preference. Each criterion maps directly to one or more of the engineering goals from the research report. They were used throughout development as a filter: any feature or design choice that could not satisfy the relevant criteria was either changed or removed.

Criteria Table
Criterion Type Measure Links to Goal
South African specificity Content All career salary, APS, subject, and demand data must be sourced from or verified against South African sources — not generic international data Goal 1, 2
Data completeness Content No career profile or bursary entry may be published with empty required fields; audit trail maintained in BURSARY-REVIEW.md Goal 1
Mobile usability Design All pages must be fully functional on a 375px viewport; minimum touch target size of 48px for all interactive elements Goal 7
Low-data performance Technical Large data files (careers, bursaries) code-split from main bundle; data saver toggle must disable animations; initial load under 200KB JS for landing page Goal 7
Free access Access No paywalls, no subscription tiers, no content behind login except progress tracking features that require a user account to function Goal 7
TVET parity Design TVET section must have equal visual prominence, equal depth of data, and equal navigation priority to university information — never a secondary page Goal 3
AI reliability Technical AI chat must have a local fallback for the 7 most common questions; must not display a blank error state; timeout handling required for all requests Goal 4
Live data currency Technical Load shedding and water dashboard data must update automatically on a schedule — not require manual commits to stay current Goal 6
Test coverage Technical All critical user flows (auth, career browse, quiz, bursary search, chat) must have a passing Playwright end-to-end test before deployment Goal 8
Accessibility Design Font sizes no smaller than 14px for body text; colour contrast meeting WCAG AA minimum; active page indicator present in all navigation contexts Goal 7
How criteria were applied in practice: During development, features that failed a criterion were not patched around — they were fixed properly or removed. The pothole reporting feature was removed in May because it failed the South African specificity criterion for the target audience: it did not directly serve a Grade 8–12 learner's education or career decisions. Community Impact was removed for the same reason.
Methodology — Section 2
02

Tools, Platforms, and Coding Environments

Every tool used in this project was chosen for a specific reason tied to the design criteria. The choices below were not arbitrary — each one was evaluated against at least one alternative before being adopted. Where alternatives were tried and discarded, this is noted.

Frontend Framework and Build Tool
Framework React 19
Component-based JavaScript framework used to build all UI in the application. Chosen for its component model, wide ecosystem, and suitability for single-page applications where server-side rendering is not needed.
Supports Goals 2, 4, 7 — interactive quiz, AI chat, mobile-responsive layouts
Build Tool Vite 6
Development server and production bundler. Chosen over Create React App for its significantly faster hot module replacement and native support for code splitting via rollupOptions — which was required to meet the low-data performance criterion.
Supports Goal 7 — careers and bursaries data files are split into separate chunks, not loaded until needed
Alternative Tried Next.js (Discarded)
The project was initially scaffolded in Next.js. It was removed after 6 days because server-side rendering added complexity with no benefit — Prospect is fully client-side. Next.js routing conventions, API routes, and deployment behavior were all overhead that the project did not need.
Discarded 28 March — see Prototype 1
Styling Tailwind CSS v4
Utility-first CSS framework. Used for all component styling. Tailwind v4 uses a Vite plugin rather than a PostCSS step, which simplified the build pipeline. The design system tokens (colors, spacing, shadows, radius) are defined in index.css and consumed throughout the app.
Supports Goal 7 — consistent responsive design across all screen sizes without separate media query stylesheets
Backend and Database
Database + Auth Supabase
Provides PostgreSQL database, authentication (email/password with session persistence), and real-time subscriptions. Used for user accounts, profile data, forum posts, saved items, and study progress tracking. Free tier is sufficient for the project's current scale.
Supports Goals 5, 8 — progress tracking persistence; auth flows covered by end-to-end tests
AI Backend Groq API
Cloud inference API for the llama-3.3-70b-versatile model. Chosen after two other AI backends failed (see AI Backend Trials in Section 4). Groq's speed — typically under 3 seconds for text responses — and free tier made it suitable for a student-facing tool. Uses the OpenAI-compatible endpoint format.
Supports Goal 4 — AI school assistant with sub-5-second response target
Deployment and Automation
Hosting Vercel
Static site and serverless deployment platform. Deploys automatically on every push to the main Git branch. Environment variables (Supabase URL, Supabase anon key, Groq API key) are stored in Vercel's encrypted environment config and injected at build time via Vite's import.meta.env system.
Supports Goal 7 — free hosting; supports Goal 6 — auto-deploys updated scraped data files
Automation GitHub Actions
CI/CD and scheduled task runner. Two scheduled workflows run automatically: one scrapes live Eskom load shedding stage data, another pulls current dam level data for the water dashboard. Both commit updated JSON files to the repository, which triggers a Vercel redeploy — keeping live data current without any manual steps.
Supports Goal 6 — live infrastructure data stays current automatically
Testing
Testing Playwright
End-to-end browser testing framework. Tests run against a live local dev server and cover all critical user journeys. A test mode bypass (?__test_mode=true) was implemented in the auth system so tests can access auth-gated pages without needing real Supabase credentials in the CI environment.
Supports Goal 8 — 31 tests passing at launch covering auth, careers, quiz, bursaries, and chat
Type Safety TypeScript
Statically typed superset of JavaScript. Used throughout the codebase. TypeScript caught several category of bugs during development that would otherwise have only appeared at runtime — particularly around the career and bursary data structures, which are large typed objects that are transformed in multiple places.
Supports Goal 8 — reduces runtime errors across the data-heavy career and bursary features
Animation and Motion
Animation Motion / React (Framer Motion)
Used for entrance animations, the AI processing status bar shimmer, page transitions, and the loading screen morphing text. All animations are disabled when the user activates the data saver toggle, and the library respects the prefers-reduced-motion media query for accessibility.
Supports Goal 7 — animations are progressive enhancement, not required for functionality
Routing React Router v6
Client-side routing library. Manages all page navigation within the single-page application. All routes are lazy-loaded — a page's JavaScript bundle is only downloaded when a user navigates to that page, rather than upfront on first load. This is a direct requirement of the low-data performance criterion.
Supports Goal 7 — lazy loading reduces initial data transfer for users on limited bundles
Methodology — Section 3
03

Procedure

The development followed a research-driven iterative process: identify a problem, establish criteria, build to meet them, test, and revise. No feature was added because it seemed like a good idea in isolation — each one had to connect to a documented user need and pass the design criteria. The procedure below covers the sequence in which major work happened and why it happened in that order.

Phase 1 — Research and Data Foundation (22–25 March)
  1. 01
    Competitor analysis
    Reviewed four existing South African platforms (Yenza, Gradesmatch, NCAP, Reslocate) across six criteria: career data, SA specificity, bursaries, TVET, study resources, and life context. Documented gaps in a comparison table. This determined what Prospect needed to do that others did not.
  2. 02
    Bursary database construction
    Sourced bursary data from university websites, government portals, and private company pages. Normalized all entries to a common schema: name, field, eligibility, amount, deadline, province. 245 entries verified and audited. A BURSARY-REVIEW.md file was maintained as an audit trail for incomplete entries.
  3. 03
    Career data cleaning
    Reviewed the initial career dataset for duplicates and inconsistencies. Removed two duplicate entries (AI/ML Engineer, Solar Installer). Defined the data schema that all 400 career profiles would follow: title, sector, salary range, required subjects, APS score, provincial demand, and grade-by-grade preparation notes.
  4. 04
    Roadmap definition
    Drafted a 7-phase development roadmap based on the competitor gaps and the data collected. Phases were prioritized by dependency: data before UI, core guidance features before community features, testing before deployment.
Phase 2 — Core Platform Build (27 March – 21 April)
  1. 05
    Framework selection and scaffold
    Next.js was tried first and removed after 6 days. React and Vite were set up as the replacement. React context was chosen over Redux for global state: the app was small enough that a simpler pattern worked, and avoiding Redux reduced bundle size and complexity.
  2. 06
    Authentication setup
    Supabase Auth wired up with email/password sign-up and sign-in. Persistent sessions enabled so users stay logged in across tabs. Human-readable error messages implemented for all auth failure states (rate limit, invalid credentials, unverified email).
  3. 07
    Career explorer, quiz, and bursary search built
    Career explorer with search and filter by subject, sector, and qualification level. Career quiz: multi-step questionnaire matching subject and interest selections to ranked career results. Bursary search with filter by field, province, and deadline status. All data loaded from typed TypeScript files; large files code-split to meet the low-data criterion.
  4. 08
    TVET section built
    Built with full navigation parity to the university section. College listings, course detail, funding options, and career outcomes all present. Navigation designed so TVET is visible at the top level — not buried as a sub-page. Confirmed against the TVET parity design criterion before sign-off.
  5. 09
    Career data expanded to 400 profiles
    A small set of 10 careers was built to the full schema first and verified in the UI before expanding. Once the data structure and display were confirmed correct, all 400 careers were built to the same standard. This prevented schema problems from only appearing after hundreds of entries had already been written.
Phase 3 — Community, AI, and Infrastructure (25 April – 15 May)
  1. 10
    Community and civic pages built
    Load shedding tracker, water dashboard, cost of living tool, NSC exam countdown, and civics guide added. Each was evaluated against the design criteria before being included — specifically whether it directly served a Grade 8–12 learner's ability to plan their education. Two features that did not meet this bar (pothole reporting, Community Impact submissions) were removed after build.
  2. 11
    Automated scrapers deployed
    GitHub Actions workflows written for load shedding and water data. Each workflow: fetches data from a public source, formats it into the required JSON schema, commits the updated file, and pushes to the repository. Vercel detects the push and redeploys automatically. Tested by verifying that data changed between scheduled runs.
  3. 12
    AI chat backend — three attempts
    Google Gemini was tried first (429 rate limit on free tier). Ollama was tried second (worked locally, failed on live site because it runs on localhost). Groq was adopted third and confirmed working end-to-end on the live URL. See Section 4 for full trial documentation.
  4. 13
    End-to-end testing suite built and verified
    31 Playwright tests written covering all critical user journeys. A test mode URL parameter (?__test_mode=true) was added to the auth system so the test suite could access protected pages without real credentials. All 31 tests confirmed passing before launch.
Methodology — Section 4
04

Prototypes, Trials, and Iterations

The platform went through six major prototype versions over 55 days, plus three separate trials for the AI backend alone. Each prototype was tested against the design criteria before the next one was started. Where a prototype failed a criterion, the decision was either to revise it or discard it — not to carry the failure forward and patch it later.

Platform Prototypes
  • P1
    Next.js Scaffold
    22–27 March 2026
    First scaffold built in Next.js with a basic landing page and signup flow. The framework added server-side rendering, API routes, and a deployment model that the project did not need. Build configuration was more complex than necessary for a fully client-side app, and the routing conventions conflicted with how the app's navigation was intended to work. Tested against the low-data criterion — the Next.js bundle structure did not allow the same level of code splitting that Vite provides.
    Discarded — replaced with React + Vite
  • P2
    React + Vite — First Build
    28–31 March 2026
    Landing page, signup, and basic study library rebuilt on React and Vite. Data saver toggle added. LocalStorage introduced for progress persistence. Accessibility audit run — font size violations corrected, touch target sizes verified, active page indicator added to header. Animation stagger delays capped for lists with 100+ items. This prototype passed the mobile usability and low-data criteria for the first time.
    Revised and carried forward
  • P3
    Full Feature Build — Three Pillars
    3–17 April 2026
    Platform restructured into three pillars (School Assist, Careers, Community). Dashboard rebuilt from a feature directory into a daily-focus view. Career data expanded to 400 profiles with full SA-specific schema. TVET section built to parity with university section. 11 UI components standardized. A full visual redesign was attempted on 17 April covering every page simultaneously — this broke the school assist layouts, the quiz, and the career card display. 30 tests written; all passing before the redesign attempt.
    Partially reverted — redesign broke 3 areas; those areas rolled back, rest kept
  • P4
    Post-Revert Stable Build
    18–27 April 2026
    The three broken areas from P3 were reverted. Images compressed to WebP format with fallback — meaningful load weight reduction on mobile, particularly on the landing page. Persistent login implemented so sessions survive tab close. New pages added: Civics, Tax and Budget Calculator, Cost of Living, School Assist Chat shell. Loading screen with animated text added. Login and signup pages redesigned to match landing page visual language. Color palette audit run — stray colors across sections brought back to the defined system.
    Kept and extended
  • P5
    Community, Scrapers, and Study Library
    1–13 May 2026
    Grade 10 Algebra learning modules built with Supabase-backed mastery tracking. Study library consolidated into one location. Community section rebuilt: Load shedding tracker and NSC exam countdown added. Automated GitHub Actions scrapers deployed for both load shedding and water data. Codebase reorganized from a flat structure into domain folders (school, careers, community, home). Pothole reporting and Community Impact submissions removed after failing the audience relevance criterion. 31 end-to-end tests confirmed passing after reorganization.
    Revised — two features removed; rest kept
  • P6
    Launch Build
    14–15 May 2026
    Groq AI backend integrated and confirmed working on the live URL. Auth requirement removed from informational community pages (load shedding, water, cost of living, civics) — these contain no user-specific data and the auth wall served no purpose. Grade 10 Subject Selector removed from landing page navigation — feature was not built to a shippable standard. Final design pass across TVET, library, calendar, and community pages. All 31 tests passing. Site redeployed and verified on the production URL.
    Launch version
AI Backend Trials
  • A1
    Google Gemini API
    13 May 2026
    The chat feature was originally built to call Google Gemini. API key configured, system prompt written for South African education context, basic integration tested locally. Tested on the live site — returned HTTP 429 (Too Many Requests). The free tier quota for the API key had been exhausted between development testing and live traffic. Gemini's free tier rate limits are per-day and cannot be reset. A local predefined answer system was built immediately as a fallback for the seven most common student questions — these return instantly without any API call.
    Discarded — rate limited, free tier insufficient
  • A2
    Ollama (Local LLM)
    15 May 2026
    Ollama is a tool for running large language models on a local machine. The llama3.2 model (2.0GB) and llava model (4.7GB, for image support) were downloaded and tested. Response quality was good and latency was acceptable on the development machine. Tested against the live site — all requests failed with a network error. Ollama runs as a server on localhost:11434. From the live Vercel deployment, every user's browser was attempting to connect to localhost on their own device, which was not running Ollama. Ollama is a local development tool; it cannot function as a cloud API backend.
    Discarded — localhost-only, not usable on hosted site
  • A3
    Groq Cloud API
    15 May 2026
    Groq provides cloud inference for the llama3 model family. The llama-3.3-70b-versatile model was selected for its balance of response quality and speed. API key generated, added to .env.local for development and to Vercel environment variables for production. Chat updated to call https://api.groq.com/openai/v1/chat/completions using the OpenAI-compatible format. System prompt configured for South African education context: NSC subjects, APS requirements, TVET pathways, matric exam structure. The predefined answer layer was kept — 7 common questions still return instantly from local storage, with Groq's response replacing them silently in the background if different. Tested end-to-end on the live URL. Working.
    Adopted — working at launch

Measurements, Variables, and Units

Software engineering does not have physical measurements in the way that chemistry or electronics do, but it has precise, quantifiable performance and quality metrics that serve the same function: they make it possible to say whether a design decision worked or not. The measurements below were used throughout development to evaluate whether the platform was meeting its design criteria.

Performance Metrics
MetricUnitTargetCriterion
Initial JS bundle size (landing page) Kilobytes (KB) Under 200 KB before gzip compression Low-data performance
AI chat response time (text) Seconds (s) Under 5 seconds from send to first token rendered AI reliability
Predefined answer response time Milliseconds (ms) Under 100ms — returned from local memory, no network call AI reliability
Minimum touch target size CSS pixels (px) 48px × 48px for all interactive elements on mobile Mobile usability, Accessibility
Minimum body font size CSS pixels (px) 14px minimum for all body and label text Accessibility
Data scraper update frequency Hours (h) Load shedding data updated at least every 6 hours via GitHub Actions Live data currency
Test suite pass rate Percentage (%) 100% — all 31 Playwright tests must pass before deployment Test coverage
Bursary database completeness Count (entries) 245 entries, all with required fields populated (verified in BURSARY-REVIEW.md) Data completeness
Career database size Count (profiles) 400 career profiles, each with full SA-specific data schema SA specificity, Data completeness
Minimum viewport support CSS pixels (px) 375px — all pages must be fully functional at this width without horizontal scroll Mobile usability
AI request timeout (text) Seconds (s) 30 seconds — AbortController cancels the request if no response within this time AI reliability
AI request timeout (image) Seconds (s) 120 seconds — image processing takes longer; extended timeout prevents false failures AI reliability
Variables

In the context of this project, the independent variable is the design and technical choices made during development — which framework was used, how data was structured, whether a feature was included or removed. The dependent variable is whether the platform meets its design criteria as measured by the metrics above. The control is the set of criteria themselves, which remained fixed from the start of development and were not changed to accommodate features that were struggling to meet them.

Note on unit type: Traditional engineering measurements (grams, newtons, volts, metres) do not apply to software development. The equivalent in this field is the combination of size, time, count, and percentage metrics listed above — all of which are equally precise, repeatable, and verifiable. The Playwright test suite provides the equivalent of a repeatable controlled experiment: the same 31 tests run against the same codebase and either pass or fail, with no ambiguity.
